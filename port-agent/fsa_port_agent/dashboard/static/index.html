What is this project?

Matching-endgame automation for The Legend of Zelda: Four Swords Adventures (GameCube, 2004) — every function in the DOL gets a row in a state DB, and this console drives the long slog of pushing each row toward a compilable, behavior-equivalent C body. End state: a browser-playable WASM build under Emscripten.

Glossary

DOL
"Dolphin" executable — GameCube's PE-equivalent, ~4.8 MB of PowerPC machine code. Lives at orig/sys/main.dol.
Matching
C that compiles byte-identical to the original binary. Gold standard for a small slice of the codebase.
Semantic
C that behaves the same but may lay out registers differently. Enough to build and run — what the browser port needs.
m2c
PowerPC → C decompiler. Its output is ugly but correct-ish; cleanup rewrites it into compilable C.
dtk
Decomp Toolkit — extracts the DOL, generates per-function asm, manages splits.txt.
Mangled name
CodeWarrior's encoded form of a C++ symbol, e.g. GetResultCam__12dCcMassS_MngCFv. Matching names across translation units drives the call graph.
Unit
A source file — one .c/.cpp owning a contiguous address range in the DOL.
Tier
Model budget for a cleanup attempt: cheap (Haiku 4.5), expensive (Sonnet 4.6), opus (Opus 4.7 on retry).

States

UNKNOWN
Not yet seen — waiting for triage.
TRIAGED
Row exists, call graph known, no source yet.
MATCHED
Byte-identical body imported from a donor source tree.
SIG_MATCHED
Signature matched via heuristics (not byte-identical).
CLEANED
Rewritten C body passed the compile gate.
BUILDS
Compiles under Emscripten end-to-end.
FAILED / PERMANENT_FAIL
Last attempt failed; retryable at a higher tier, then parked.

Where does the data come from?

  • port-agent/state.db — SQLite, one row per DOL function.
  • port-agent/work/ — prompt↔response filesystem queue.
  • orig/sys/main.dol — extracted from your RVZ disc image.
