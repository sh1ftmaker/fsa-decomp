What is this page?

This dashboard tracks an automated effort to rebuild The Legend of Zelda: Four Swords Adventures (FSA, GameCube 2004) from its compiled game disc back into readable C/C++ source code, then re-compile that source for the web browser. The rebuild happens one function at a time, across 5,981 individual functions in the game's executable.

If you have never heard of "decompilation" before, the What is decompilation? section below has a from-scratch primer. If you already know, skip to Gate 4 for the current status.

① What is decompilation?

When Nintendo shipped FSA in 2004, they started with a source tree of C++ files — human-readable code like if (player.hearts < 0) die(); — and ran it through a compiler (Metrowerks CodeWarrior) to produce a binary: a 4.8 MB file of PowerPC machine instructions (main.dol) that the GameCube CPU knows how to execute directly.

The source was never published. The binary is all we have. Decompilation is the process of reversing the compilation step — turning the binary back into something that looks like, and ideally compiles back to, the original source.

There are two flavors, and this project uses both:

Matching decomp

Write C/C++ that compiles to byte-identical output to the original binary. This is the gold standard — it proves the source is semantically and textually a faithful reproduction. It's also extremely fiddly: you have to guess the original compiler version, optimization flags, and sometimes even the order the developer declared local variables.

Status in FSA: ~7.2% of functions byte-match.

Semantic decomp (browser port)

Write C/C++ that behaves the same but doesn't need to compile to identical bytes. This is what we need to run the game in a browser via WebAssembly — we don't care about PowerPC register allocation, we just need the logic to be correct.

The five-phase pipeline is doing this for all 5,981 functions.

Why The Wind Waker helps. FSA was developed shortly after, and shares ~80% of its codebase with, The Legend of Zelda: The Wind Waker (TWW, 2002). TWW has already been fully decompiled by the zeldaret team. When a function in FSA and a function in TWW share the same mangled name (the CodeWarrior compiler's encoded form of ClassName::methodName(args)), we can often copy TWW's source over and use it as a nearly-correct starting point.

② The five-phase pipeline

Rebuilding every function by hand would take years. Instead, the port agent runs a staged pipeline, most of it automated, with a large language model (Claude Code) doing the human-like cleanup steps. Here's each phase in plain terms:

  1. Triage — read the binary's 5,981 functions, record each one in a database, and map out which function calls which. Output is a call graph and some tags (LEAF for functions that don't call anything, VTABLE_THUNK for C++ virtual dispatch, etc.). Fast — seconds.
  2. TWW mass import — for every source file in TWW, compile it with FSA's compiler flags and search the FSA binary for matching byte patterns. Every hit means we got a "free" function — the code is already written, just needs an address recorded. Slow — hours.
  3. Bottom-up cleanup — for each remaining function, give Claude Code the raw decompiler output (from a tool called m2c), the function's callees and callers, any matching TWW source, and ask it to produce compilable C. Processed leaf-first so by the time we get to main, every callee is already done. The biggest AI workload.
  4. HAL scaffold — the GameCube hardware doesn't exist in a browser. Stub out the hardware-abstraction layer (GPU, audio, disc, pad, threading) with Emscripten-compatible shims. Scripted, ~seconds.
  5. Build loop — try to compile the whole thing. For each error, hand Claude Code the error + surrounding code, get a patch, apply it, try again. Iterate until the binary builds. AI loop — many rounds.

Each function in the database moves through states as phases run:

UNKNOWN — not yet seen
TRIAGED — in database, call graph known
MATCHED_TWW — byte-identical copy from TWW
SIG_MATCHED — signature matched via heuristics
CLEANED — m2c output rewritten to compilable C
BUILDS — compiles successfully under Emscripten
FAILED — tried, didn't work (retryable)

③ Gate 4 — TWW mass-import coverage

Gate 4 is the project's go/no-go: before investing in phases 3–5, we need at least 20% of the DOL's functions to have been copied in from TWW automatically. The bar below shows how close we are. The orange tick mark is the 20% threshold.
0 matched via TWW / 5981 DOL fns (0%) — need 1197 to pass

The ceiling turns out to be around 7% — the remaining 73% of FSA code is game-specific content (new enemies, new dungeon scripts) that can't byte-match against TWW. Those functions go through phase 3 cleanup instead, where TWW source still helps as a naming/types reference even when bytes don't line up.

⑥ Work queue

When Claude Code needs to rewrite a function (phase 3) or fix a build error (phase 5), the agent writes a Markdown prompt file to port-agent/work/<kind>/ and waits for a sibling response file to appear. "Pending" is prompts waiting for AI; "responses" is prompts that got an AI answer and are waiting for --apply to splice them back into the source tree. "Archived" is completed triplets (prompt+response+meta) moved to done/.

Where does the data come from?

  • port-agent/state.db — SQLite. One row per DOL function, one row per call-graph edge, one row per string reference. All the charts above query this.
  • config/G4SE01/splits.txt — dtk's ground-truth mapping of DOL address ranges to source files.
  • config/G4SE01/symbols.txt — symbol names & sizes. FSA's copy uses TWW's name for any function that shares a mangled symbol.
  • orig/sys/main.dol — the GameCube executable, extracted from your RVZ disc image via dtk disc extract. Not committed (copyrighted).
  • ~/Desktop/tww/ — sibling clone of zeldaret/tww. Donor source tree for the Gate 4 import and for cleanup references.

What should I click first?

On a fresh checkout: Run Triage → TWW Import (dry-run) to see the gate reading → TWW Import (real) if you have hours to spare. After that, batches of Prepare 10 cleanup drive the AI cleanup loop.
